# Bigger Applications — Multiple Files

As your application grows you will spread logic across many files. Nitro uses the same
**Handlers / Routes** separation that Django popularised: view-logic lives in handler
modules, URL declarations live in a single routes file.

## Recommended project layout

```
app/
├── Project.toml
├── Manifest.toml
├── db/
│   └── automatic_models.jl   # generated by PormG
└── src/
    ├── App.jl           # top-level module: loads PormG, includes everything
    ├── Routes.jl        # all path() declarations
    └── Handlers/
        ├── ProductHandlers.jl
        └── OrderHandlers.jl
```

The `src/` tree is a Julia module. `App.jl` is the entry-point: it loads PormG,
imports the generated model module, includes all handlers and routes, then calls `serve()`.

---

## Setting up PormG models

PormG models are generated from your database schema and stored outside `src/` (e.g. `db/`).
Load them in the top-level module with `@import_models`, which creates a plain Julia module
you can alias and pass down to handlers via `import`.

```julia
# src/App.jl
module App

using Nitro
using HTTP
using PormG

# Point PormG at your database config directory
PormG.Configuration.load_many(["db"])

# Import the generated model file — creates a module named `automatic_models`
PormG.@import_models "../db/automatic_models.jl" automatic_models

# Alias for convenience: handlers do `import ..appM`
const appM = automatic_models

include("Handlers/ProductHandlers.jl")
include("Handlers/OrderHandlers.jl")
include("Routes.jl")

using .Routes

end # module
```

Handlers import the alias and use `appM.Product.objects` for queries:

```julia
# src/Handlers/ProductHandlers.jl
module ProductHandlers

using HTTP
using Nitro
using PormG
import ..appM   # the alias defined in App.jl

export list_products, get_product

function list_products(req::HTTP.Request)
    query = appM.Product.objects
    return Res.json(list(query))
end

function get_product(req::HTTP.Request, id::Int)
    query = appM.Product.objects
    query.filter("id" => id)
    result = list(query)
    isempty(result) && return Res.status(404, "not found")
    return Res.json(first(result))
end

end # module
```

## Defining handlers

Handler functions live in plain Julia modules — one file per domain area.

```julia
# src/Handlers/MathHandlers.jl
module MathHandlers

using HTTP
using Nitro

export multiply, divide

function multiply(req::HTTP.Request, a::Float64, b::Float64)
    return Res.json(a * b)
end

function divide(req::HTTP.Request, a::Float64, b::Float64)
    a == 0 && return Res.status(400, "divisor cannot be zero")
    return Res.json(a / b)
end

end # module
```

## Declaring routes

All `path()` calls are collected in `Routes.jl`.  This is the only place that
wires handler functions to URL patterns.

```julia
# src/Routes.jl
using Nitro
import .MathHandlers
import .ProductHandlers

const urlpatterns = [
    path("/api/math/multiply/<float:a>/<float:b>", MathHandlers.multiply, method="GET"),
    path("/api/math/divide/<float:a>/<float:b>",   MathHandlers.divide,   method="GET"),

    path("/api/products",               ProductHandlers.list_products,  method="GET"),
    path("/api/products/<int:id>",      ProductHandlers.get_product,    method="GET"),
    path("/api/products",               ProductHandlers.create_product, method="POST"),
]
```

## Splitting routes across files with `include_routes`

For larger apps, split `urlpatterns` into per-domain files and compose them with
`include_routes`.

```
src/
└── Routes/
    ├── math_routes.jl
    └── product_routes.jl
```

```julia
# src/Routes/math_routes.jl
using Nitro
import ..MathHandlers

const math_routes = [
    path("/multiply/<float:a>/<float:b>", MathHandlers.multiply, method="GET"),
    path("/divide/<float:a>/<float:b>",   MathHandlers.divide,   method="GET"),
]
```

```julia
# src/Routes.jl
using Nitro
include("Routes/math_routes.jl")
include("Routes/product_routes.jl")

const urlpatterns = [
    include_routes("/api/math",    math_routes),
    include_routes("/api/products", product_routes),
]
```

The prefix passed to `include_routes` is prepended to every path in the included list,
so `/multiply/<float:a>/<float:b>` becomes `/api/math/multiply/<float:a>/<float:b>`.

## Wiring everything in `main.jl`

```julia
# src/App.jl  (or main.jl for simpler projects)
using Nitro

include("Handlers/ProductHandlers.jl")
include("Handlers/OrderHandlers.jl")
include("Routes.jl")

serve(urlpatterns, middleware=[SessionMiddleware, RateLimiter])
```

`serve()` accepts the `urlpatterns` vector and any global middleware. Global
middleware runs before every route, regardless of which handler it targets.

---

## Middleware layers

Nitro supports middleware at two levels: **global** (via `serve()`) and
**route-level** (via the `middleware=` keyword on `path()`).

Execution order is always:

```
global middleware → route middleware → handler
```

### Global middleware

```julia
serve(urlpatterns, middleware=[SessionMiddleware, RateLimiter])
```

### Route-level middleware

Apply middleware to individual routes by passing a `middleware=` vector to `path()`.

```julia
const urlpatterns = [
    # public — no extra middleware
    path("/api/products", ProductHandlers.list_products, method="GET"),

    # protected — requires authentication
    path("/api/products", ProductHandlers.create_product,
         method="POST", middleware=[login_required]),

    path("/api/products/<int:id>", ProductHandlers.delete_product,
         method="DELETE", middleware=[login_required, role_required("admin")]),
]
```

### Bypassing global middleware on a route

Pass `middleware=[]` on a `path()` call to opt that route out of **all** middleware
(including the global ones).  Useful for health-check endpoints that must always
respond, even when the session store is unavailable.

```julia
path("/health", HealthHandlers.ping, method="GET", middleware=[])
```

## Internal requests

Use `internalrequest` to call one of your own endpoints from within another
handler — for example, to reuse validation logic or chain computations:

```julia
using HTTP

function process_and_fetch(req::HTTP.Request, id::Int)
    # call another endpoint inside the same server
    inner = internalrequest(HTTP.Request("GET", "/api/products/$id"))
    data  = JSON3.read(inner.body)
    # ... further processing
    return Res.json(data)
end
```
